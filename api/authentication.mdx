## Overview

Khedma Market uses [NextAuth.js](https://next-auth.js.org) for authentication, integrated seamlessly with tRPC. The authentication system supports multiple providers and includes features like two-factor authentication and email verification.

## Authentication providers

Khedma Market supports three authentication methods:

### OAuth providers

<CodeGroup>
```typescript GitHub
GithubProvider({
  clientId: env.GITHUB_CLIENT_ID,
  clientSecret: env.GITHUB_CLIENT_SECRET,
  allowDangerousEmailAccountLinking: true,
})
```

```typescript Google
GoogleProvider({
  clientId: env.GOOGLE_CLIENT_ID,
  clientSecret: env.GOOGLE_CLIENT_SECRET,
  allowDangerousEmailAccountLinking: true,
})
```
</CodeGroup>

### Credentials (Email and Password)

```typescript
Credentials({
  async authorize(credentials) {
    const validatedFields = LoginSchema.safeParse(credentials);
    
    if (validatedFields.success) {
      const { email, password } = validatedFields.data;
      const user = await getUserByEmail(email);
      
      if (!user || !user.password) return null;
      
      const passwordsMatch = await bcrypt.compare(password, user.password);
      if (passwordsMatch) return user;
    }
    
    return null;
  },
})
```

<Warning>
  Credentials-based sign-in requires email verification. OAuth providers bypass this requirement.
</Warning>

## Session management

### Session structure

The session object extends NextAuth's default session with additional user properties:

```typescript
export type ExtendedUser = DefaultSession["user"] & {
  id: string;
  role: "client" | "freelancer" | "company";
  username: string;
  isTwoFactorEnabled: boolean;
  isOAuth: boolean;
};

declare module "next-auth" {
  interface Session {
    user: ExtendedUser;
  }
}
```

### Retrieving session

<CodeGroup>
```typescript Server Components
import { getServerAuthSession } from "@/server/auth";

export default async function ProfilePage() {
  const session = await getServerAuthSession();
  
  if (!session) {
    redirect("/auth/sign-in");
  }
  
  return (
    <div>
      <h1>Welcome, {session.user.name}!</h1>
      <p>Role: {session.user.role}</p>
      <p>Username: {session.user.username}</p>
    </div>
  );
}
```

```tsx Client Components
import { useSession } from "next-auth/react";

export function UserMenu() {
  const { data: session, status } = useSession();
  
  if (status === "loading") {
    return <div>Loading...</div>;
  }
  
  if (!session) {
    return <a href="/auth/sign-in">Sign In</a>;
  }
  
  return (
    <div>
      <p>{session.user.name}</p>
      <p>Role: {session.user.role}</p>
    </div>
  );
}
```
</CodeGroup>

## Protected procedures

tRPC provides two types of procedures for handling authentication:

### Public procedures

Public procedures can be called by anyone, but still have access to session data:

```typescript
import { publicProcedure, createTRPCRouter } from "@/server/api/trpc";

export const profileRouter = createTRPCRouter({
  getSkills: publicProcedure.query(async ({ ctx }) => {
    // ctx.session may be null or contain user data
    return await ctx.db.skill.findMany({
      orderBy: { name: "asc" },
    });
  }),
});
```

### Protected procedures

Protected procedures require authentication and guarantee that `ctx.session.user` exists:

```typescript
import { protectedProcedure, createTRPCRouter } from "@/server/api/trpc";
import { TRPCError } from "@trpc/server";

export const protectedProcedure = t.procedure.use(({ ctx, next }) => {
  if (!ctx.session || !ctx.session.user) {
    throw new TRPCError({ code: "UNAUTHORIZED" });
  }

  return next({
    ctx: {
      session: { ...ctx.session, user: ctx.session.user },
    },
  });
});
```

<Info>
  When you use `protectedProcedure`, TypeScript knows that `ctx.session.user` is never null.
</Info>

### Using protected procedures

<CodeGroup>
```typescript Query Example
export const profileRouter = createTRPCRouter({
  getUserSkills: protectedProcedure.query(async ({ ctx }) => {
    // ctx.session.user is guaranteed to exist
    return await ctx.db.skill.findMany({
      where: { userId: ctx.session.user.id },
      select: {
        id: true,
        name: true,
        value: true,
        level: true,
      },
    });
  }),
});
```

```typescript Mutation Example
export const userRouter = createTRPCRouter({
  updateUserPersonalInfo: protectedProcedure
    .input(personalFormSchema)
    .mutation(async ({ input, ctx }) => {
      // ctx.session.user.id is type-safe
      return await ctx.db.user.update({
        where: { id: ctx.session.user.id },
        data: { ...input },
      });
    }),
});
```
</CodeGroup>

## Client-side authentication

### Calling protected procedures

When calling protected procedures from the client, include credentials:

<CodeGroup>
```tsx React Query Hook
import { api } from "@/trpc/react";

export function UserSkills() {
  const { data, isLoading, error } = api.profile.getUserSkills.useQuery();
  
  if (error?.data?.code === "UNAUTHORIZED") {
    return <div>Please sign in to view your skills</div>;
  }
  
  if (isLoading) return <div>Loading...</div>;
  
  return (
    <ul>
      {data?.map((skill) => (
        <li key={skill.id}>
          {skill.name} - {skill.level}
        </li>
      ))}
    </ul>
  );
}
```

```tsx Mutation Hook
import { api } from "@/trpc/react";

export function AddSkillForm() {
  const utils = api.useUtils();
  
  const addSkill = api.profile.addSkill.useMutation({
    onSuccess: () => {
      // Invalidate and refetch skills
      utils.profile.getUserSkills.invalidate();
    },
    onError: (error) => {
      if (error.data?.code === "UNAUTHORIZED") {
        router.push("/auth/sign-in");
      }
    },
  });
  
  const handleSubmit = (values: { name: string; level: string }) => {
    addSkill.mutate(values);
  };
  
  return <form onSubmit={handleSubmit}>{/* form fields */}</form>;
}
```
</CodeGroup>

## Two-factor authentication

Khedma Market supports optional two-factor authentication:

### Enabling 2FA

```typescript
export const userRouter = createTRPCRouter({
  updateUserSecurityInfo: protectedProcedure
    .input(accountFormSchema)
    .mutation(async ({ input, ctx }) => {
      return await ctx.db.user.update({
        where: { id: ctx.session.user.id },
        data: {
          isTwoFactorEnabled: input.isTwoFactorEnabled,
        },
      });
    }),
});
```

### 2FA sign-in flow

```typescript
async signIn({ user, account }) {
  // Allow OAuth without email verification
  if (account?.provider !== "credentials") return true;

  const existingUser = await getUserById(user.id);

  // Prevent sign in without email verification
  if (!existingUser?.emailVerified) return false;

  if (existingUser.isTwoFactorEnabled) {
    const twoFactorConfirmation = await getTwoFactorConfirmationByUserId(
      existingUser.id,
    );

    if (!twoFactorConfirmation) return false;

    // Delete two factor confirmation for next sign in
    await db.twoFactorConfirmation.delete({
      where: { id: twoFactorConfirmation.id },
    });
  }

  return true;
}
```

<Note>
  Two-factor authentication is only required for credentials-based sign-in, not OAuth.
</Note>

## Authorization by role

Check user roles in protected procedures:

```typescript
export const userRouter = createTRPCRouter({
  createCompany: protectedProcedure
    .input(companySchema)
    .mutation(async ({ ctx, input }) => {
      const user = await ctx.db.user.findFirst({
        where: { id: ctx.session.user.id },
        select: { id: true, role: true },
      });

      if (user?.role !== "company") {
        throw new TRPCError({
          code: "FORBIDDEN",
          message: "User is not signed up as a company",
        });
      }

      return await ctx.db.company.create({
        data: {
          ...input,
          userId: user.id,
        },
      });
    }),
});
```

## Session callbacks

NextAuth callbacks enrich the session with user data:

```typescript
callbacks: {
  async session({ token, session }) {
    if (token.sub && session.user) {
      session.user.id = token.sub;
    }

    if (token.role && session.user) {
      session.user.role = token.role as role;
    }

    if (session.user) {
      session.user.isTwoFactorEnabled = token.isTwoFactorEnabled as boolean;
      session.user.name = token.name;
      session.user.username = token.username as string;
      session.user.email = token.email;
      session.user.isOAuth = token.isOAuth as boolean;
    }

    return session;
  },
  
  async jwt({ token }) {
    if (!token.sub) return token;

    const existingUser = await getUserById(token.sub);
    if (!existingUser) return token;
    
    const existingAccount = await getAccountByUserId(existingUser.id);
    
    token.name = existingUser.name;
    token.username = existingUser.username;
    token.email = existingUser.email;
    token.role = existingUser.role;
    token.isOAuth = !!existingAccount;
    token.isTwoFactorEnabled = existingUser.isTwoFactorEnabled;

    return token;
  },
}
```

## Best practices

<AccordionGroup>
  <Accordion title="Always use protectedProcedure for user-specific data">
    Never rely on checking `ctx.session` manually. Use `protectedProcedure` to guarantee authentication and get proper TypeScript types.
    
    ```typescript
    // Good
    protectedProcedure.query(async ({ ctx }) => {
      return ctx.db.user.findUnique({ 
        where: { id: ctx.session.user.id } 
      });
    });
    
    // Bad
    publicProcedure.query(async ({ ctx }) => {
      if (!ctx.session?.user) throw new Error("Unauthorized");
      return ctx.db.user.findUnique({ 
        where: { id: ctx.session.user.id } // TypeScript error!
      });
    });
    ```
  </Accordion>
  
  <Accordion title="Handle UNAUTHORIZED errors on the client">
    Always handle unauthorized errors gracefully and redirect users to sign in:
    
    ```tsx
    const mutation = api.user.updateProfile.useMutation({
      onError: (error) => {
        if (error.data?.code === "UNAUTHORIZED") {
          router.push("/auth/sign-in");
        }
      },
    });
    ```
  </Accordion>
  
  <Accordion title="Use role-based authorization when needed">
    For endpoints that require specific roles, check the user's role explicitly:
    
    ```typescript
    protectedProcedure.mutation(async ({ ctx }) => {
      if (ctx.session.user.role !== "company") {
        throw new TRPCError({ code: "FORBIDDEN" });
      }
      // Company-only logic
    });
    ```
  </Accordion>
  
  <Accordion title="Secure OAuth with email verification">
    For credentials-based auth, always verify emails before allowing sign-in. OAuth providers are trusted and skip this check.
  </Accordion>
</AccordionGroup>
