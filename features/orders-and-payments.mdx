## Creating an order

Orders are created when a client purchases a gig:

### Selecting a package

1. Browse gigs and find a service
2. View gig details and packages
3. Select your desired package:
   - Basic: Core service
   - Standard: Enhanced features
   - Premium: Complete package
4. Review package details:
   - Price
   - Delivery time
   - Number of revisions
   - What's included

### Order creation process

When you click "Order Now":

```typescript
model Order {
  id          String      @id @default(cuid())
  userId      String      // Buyer
  gigId       String      // Service ordered
  packageId   String      // Selected package
  status      OrderStatus @default(pending)
  totalAmount Float       // Price in MAD
  createdAt   DateTime    @default(now())
  updatedAt   DateTime    @updatedAt
  user        User        @relation(fields: [userId], references: [id])
  gig         Gig         @relation(fields: [gigId], references: [id])
  package     Package     @relation(fields: [packageId], references: [id])
  payment     Payment?
}
```

The order captures:
- Which user is buying
- Which gig and package
- Total amount
- Initial status (pending)

<Info>
  Orders remain in "pending" status until payment is confirmed.
</Info>

## Payment processing

Khedma Market uses YouCan Pay for secure payment processing:

### Payment flow

1. **Order creation**: Order is created with status "pending"
2. **Payment tokenization**: Amount is tokenized for secure processing
3. **Payment page**: User is redirected to payment gateway
4. **Payment completion**: User completes payment
5. **Confirmation**: Platform receives payment confirmation
6. **Order update**: Order status changes to "processing"

```typescript
model Payment {
  id            String        @id @default(cuid())
  orderId       String        @unique
  status        PaymentStatus @default(pending)
  amount        Float
  paymentMethod String        // Payment method used
  transactionId String?       // Gateway transaction ID
  createdAt     DateTime      @default(now())
  updatedAt     DateTime      @updatedAt
  order         Order         @relation(fields: [orderId], references: [id])
}
```

### Payment statuses

```typescript
enum PaymentStatus {
  pending   // Awaiting payment
  success   // Payment confirmed
  failed    // Payment failed
}
```

### Payment tokenization

Before redirecting to payment:

```typescript
const tokenizePayment = async (
  amount: number,
  currency: string,
  orderId: string
) => {
  const formData = new FormData();
  formData.append('amount', amount.toString());
  formData.append('currency', currency);
  formData.append('pri_key', process.env.PRIVATE_YOUCAN_KEY);
  formData.append('order_id', orderId);
  
  const response = await youcan.post('/tokenize', formData);
  
  return response.data.token.id;
};
```

This creates a secure payment token used for processing.

<Note>
  Payment details are handled securely by the payment gateway. Khedma Market never stores sensitive payment information.
</Note>

## Delivery and completion

### Delivering work

As a seller, deliver completed work:

1. Prepare final deliverables
2. Upload files or provide access
3. Send delivery notification to buyer
4. Await buyer confirmation

**Delivery methods:**
- Upload files through messaging
- Share download links
- Provide access to online resources
- Send completed work as message attachments

### Requesting revisions

Buyers can request changes:

- Revisions included in package:
  - Basic: Limited revisions
  - Standard: More revisions
  - Premium: Maximum revisions

- Revision workflow:
  1. Buyer reviews delivered work
  2. Requests specific changes via message
  3. Seller makes revisions
  4. Delivers updated version
  5. Process repeats until approved or revisions exhausted

<Info>
  Number of revisions is defined in the package. Additional revisions may require a new order or custom arrangement.
</Info>

### Marking orders complete

Orders are completed when:

1. **Buyer approval**: Buyer reviews and accepts the work
2. **Auto-completion**: Order auto-completes after delivery if no action taken (platform-specific timeline)
3. **Mutual agreement**: Both parties agree work is complete

```typescript
const completeOrder = async (orderId: string) => {
  return await db.order.update({
    where: { id: orderId },
    data: { status: 'completed' }
  });
};
```

## Order cancellation

Orders can be cancelled under certain conditions:

### Before work begins

- Buyer can request cancellation
- Seller can accept or decline
- If accepted, refund is processed

### After work begins

- Mutual agreement required
- Partial payment may apply based on work completed
- Platform support can mediate disputes

### Cancellation process

1. One party requests cancellation
2. Other party is notified
3. Cancellation reason provided
4. If accepted, order status changes to "cancelled"
5. Refund processed according to policy

```typescript
const cancelOrder = async (orderId: string) => {
  await db.order.update({
    where: { id: orderId },
    data: { status: 'cancelled' }
  });
  
  // Process refund if payment was made
  const payment = await db.payment.findUnique({
    where: { orderId }
  });
  
  if (payment?.status === 'success') {
    // Initiate refund process
  }
};
```

<Note>
  Cancellation policies protect both buyers and sellers. Review the platform's cancellation policy before requesting cancellation.
</Note>
